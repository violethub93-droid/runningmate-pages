# 러닝메이트 — 기능 로드맵

> 혼자 뛰는 사람을 위한 실시간 AI 음성 메이트
> 핵심 차별점: 측정이 아니라 **내 상태에 실시간 반응하고, 내가 고른 목소리로 함께 뛰는 존재**

---

## 설계 원칙 한 줄

측정·기록은 Strava·Nike Run Club이 이미 잘한다. "초보용 음성 코칭"은 런데이가 이미 점유했다. 러닝메이트가 이기는 지점은 **"내 상태에 실시간으로 반응하고, 내가 고른 목소리로, 내 케이던스에 맞춰 함께 뛰는가"** — 즉 적응형 코칭 + 페르소나 선택 + 케이던스 동기화다. 모든 우선순위는 이 기준으로 정렬한다.

---

## 경쟁 포지셔닝 — vs 런데이

런데이는 "혼자 뛰는 초보 러너를 위한 음성 코칭"이라는 콘셉트를 한국 시장에서 이미 점유한 직접 경쟁자다. 정면으로 같은 간판을 걸면 후발주자로 불리하다. 차별화는 런데이가 **구조적으로 약한 축**에서만 가능하며, 그 축이 곧 우리의 핵심 결정들과 일치한다.

| 축 | 런데이 | 러닝메이트 (차별화 쐐기) |
|----|--------|------------------------|
| 코칭 방식 | **프로그램 기반** (24일 코스 등 정해진 플랜의 타이밍 멘트) | **실시간 적응형** (지금 내 페이스·케이던스 이탈에 즉시 반응) |
| 목소리 | 단일 '아저씨' 풀보이스 녹음 — *부담된다는 후기 다수* | **페르소나 선택** (코치/친구/동기부여), TTS 캐싱으로 확장 용이 |
| 음악 | 강도별 자동 선곡 + 멜론 연계 | **케이던스 BPM 동기화** (보행을 끌어올리는 코칭 도구) |
| 종료 후 | 스탬프·기록 | **생성형 AI 서사형 리포트** |

**정면 대결을 피할 영역** (런데이의 본진, 소모전 주의):
- 커뮤니티·크루·친구 음성 응원 등 소셜 — 네트워크 효과로 선점됨. 게다가 "혼자 뛰는 사람" 콘셉트와 충돌.
- 24일 PT형 구조화 훈련 플랜 — 런데이의 핵심 자산.

> 한 줄 요약: **"초보용 음성 코칭 앱"으로는 못 이긴다(그건 런데이다). "실시간 적응형 + 내가 고른 메이트"로 이긴다.**

## 1차 현장 테스트 결과 (검증 완료)

클로바더빙 무료 버전으로 코치형 음성을 합성해 실제로 달리며 검증. **방식: 수동 재생**(자동 트리거 미적용).

**✅ MVP 핵심 가설 통과**
- "동행자처럼 느껴졌는가, 트래커 음성안내처럼 느껴졌는가?" → **동행자처럼 느껴짐.** 콘셉트 검증 성공.
- 단, "특별한 경험"으로는 부족 — 음성 톤·어조가 실제 녹음만큼 자연스럽지 않음(실행 품질 과제).

**핵심 발견 (실행 과제로 전환)**
1. **GPS 페이스 노이즈 ↔ 지연의 긴장.** 계단 한 층만 올라도 페이스가 급등. 노이즈를 평균으로 누르면 코칭이 늦어져 *적막*해지고, 빠르게 반응하면 계단마다 오발사. → 해법: **중앙값(median) 필터 + 경사 보정 + 케이던스/기압계 활용.**
2. **적막함 문제 — 기존 가정이 뒤집힘.** 우려했던 "잔소리"가 아니라, 오히려 **너무 조용해서 혼자 같았음.** 페이스 변화 타이밍에 멘트가 늦게/적게 나옴. → cooldown·트리거 기본값을 *존재감을 더 채우는 방향*으로 재설정.
3. **음성 자연스러움 부족.** 내장 TTS·런데이보다는 낫지만 "특별함"엔 미달. → 합성 품질 업그레이드 또는 핵심 멘트 성우 녹음 하이브리드.
4. **문구 풀이 작아** 빈도·반복·적합성 판단 불가. → 다음 테스트 전 코치형·친구형 풀 확장 + 자동 트리거 필요.

> 결론: **방향은 맞았다(동행자 확인). 이제 콘셉트 문제가 아니라 완성도 문제다.** 우선순위가 아래처럼 재조정됨 — 페이스 스무딩·선제 코칭을 앞당기고, cooldown 가정을 수정.

## 단계별 로드맵

### Phase 0 — MVP (검증용 최소 단위)

> 목표: "옆에서 같이 뛰어준다"는 핵심 마법이 전달되는지만 확인한다.

| 기능 | 왜 지금 | 개발 난이도 |
|------|---------|------------|
| GPS 기반 실시간 페이스 추적 | 코칭의 유일한 입력 재료 | 낮음 (표준 라이브러리) |
| **페이스 스무딩 (중앙값 필터)** | **★현장 발견: 계단·노이즈 스파이크 제거 — 없으면 오발사** | 낮음~중간 |
| 목표 페이스 설정 | 코칭 기준선 | 낮음 |
| 페이스 이탈 시 음성 코칭 1줄 | **이게 곧 "메이트"** | 중간 (규칙 + 캐싱 TTS) |
| 종료 후 AI 요약 리포트 | 다시 켜게 만드는 동기부여 | 중간 (생성형 AI, 텍스트) |
| 거리·시간·평균 페이스 기록 | 위 데이터의 자연스러운 결과물 | 낮음 |

> ★ 1차 현장 테스트로 추가됨. GPS 원시 페이스는 계단 한 층에도 급등하므로, 중앙값 필터가 MVP 필수 요소로 승격.

**확정된 기본값 (이 결정으로 MVP 범위가 고정됨):**
- **타깃**: 초보 러너 우선 → 코칭은 관대하게(페이스 숫자 집착 X), 부상예방·습관화 중심
- **페르소나 기본값**: 차분한 코치형 (단, 데이터 구조는 처음부터 다중 페르소나 대비해 설계 — 아래 별도 섹션)
- **음성**: 합성 TTS, 단 **사전 합성 후 음성파일 캐싱** 방식 (실시간 호출 X)

MVP에서 의도적으로 **빼는 것**: 보폭·자세 교정, 경사 분석, 코스 추천, 소셜 기능. 핵심이 검증되기 전엔 붙이지 않는다.

### Phase 1 — 코칭 깊이 더하기

> 목표: 음성 코치를 "한 줄짜리"에서 "진짜 코치"로.

| 기능 | 왜 다음인가 | 개발 난이도 |
|------|------------|------------|
| 가속도계(G센서) 케이던스·보폭 추정 | 페이스 외 두 번째 코칭 축 | 중간 |
| 케이던스 기반 자세 코칭 | 오버스트라이드 교정 = 부상 예방 | 중간 |
| 발화 타이밍 정교화 (이탈·체크인·마일스톤 트리거) | ★현장 발견: "잔소리"가 아니라 "적막함"이 문제 — 존재감 채우는 방향으로 | 중간 |
| 코칭 없는 구간 BGM + 덕킹 | 메이트 말할 때만 줄고, 평소 깔린다 | 중간 |
| 음성 문구 풀 다양화 (생성형 AI로 사전 생성) | 로봇처럼 반복되지 않게 | 낮음~중간 |
| 페르소나 데이터 구조 확정 (상황 × 페르소나) | 나중 페르소나 추가를 데이터 작업으로 | 낮음 |

---

### Phase 2 — 지형·코스 지능

> 목표: "어디를 뛰는가"까지 코칭에 반영.

| 기능 | 왜 이 단계인가 | 개발 난이도 |
|------|--------------|------------|
| 고도 데이터(DEM/Elevation API) 기반 경사 분석 | 오르막/내리막별 코칭 | 중간~높음 |
| 경사 보정 페이스(GAP) | ★현장 발견으로 **우선순위 상승** — 계단·오르막 페이스 급등 오발사 방지 | 높음 |
| **선제적 고도 코칭 (오르막 진입 전 예고)** | ★지연 약점을 리드타임으로 전환 — "곧 오르막이에요, 호흡 준비". 런데이 불가 | 중간~높음 |
| 러닝 코스·구간 목표 설정 | 사용자가 처음 구상한 기능 | 중간 |
| 오프라인 GPS 보정 (음영지역 대응) | 산·터널에서 끊김 방지 | 높음 |
| 일일/주간 운동량 누적 대시보드 | 습관화 유도 | 낮음 |

> ★ 1차 현장 테스트 반영: GPS 페이스가 경사에 크게 흔들려, 경사 보정과 선제 고도 코칭의 우선순위가 올라감. 기압계(barometer)로 고도를 잡으면 GPS 고도보다 안정적.

> 주의: 처음 구상한 "HD맵"은 자율주행용 정밀지도라 러닝 앱엔 과하고 비싸다. 실제로 필요한 건 **고도 데이터(Elevation API)** 이며, 이걸로 경사 코칭이 충분히 가능하다.

### Phase 3 — 확장

> 목표: 리텐션·차별화 강화 (핵심 검증 이후에만).

| 기능 | 비고 | 개발 난이도 |
|------|------|------------|
| 페르소나 확장 (친구형·동기부여형 등) | 어투·톤·단어 다양성 — 런데이 단일보이스 대비 우위 | 중간 (구조는 Phase 1에 마련) |
| 케이던스 BPM 동기화 BGM | 현재보다 살짝 높은 BPM으로 케이던스 유도 = 은근한 코치 | 높음 |
| 중급자 모드 (스플릿·목표 레이스 페이스·GAP) | 초보가 성장해도 떠나지 않게 | 중간~높음 |
| 코스 추천 / 인기 구간 | 데이터 축적 필요 | 중간 |
| 실시간 대화형 메이트 (음성 응답) | 기술·배터리·지연 난이도 높음, 신중히 | 매우 높음 |

## 우선순위 × 난이도 매트릭스

```
난이도 ↑
높음 │              GAP·경사코칭        실시간 대화형 메이트
     │              오프라인 GPS보정    음악 동기화
     │
중간 │   음성코칭 1줄   케이던스 코칭     코스 목표설정
     │   AI 요약 리포트  타이밍 정교화
     │
낮음 │   GPS 페이스     문구풀 다양화     운동량 대시보드
     │   목표 설정      메이트 톤 선택
     └──────────────────────────────────────────────→
        지금 (MVP)      다음              나중      우선순위
```

좌측 하단(낮은 난이도 · 높은 우선순위)부터 처리하고, 우측 상단(높은 난이도)은 핵심 검증 후로 미룬다.

## 오디오 설계 (TTS + BGM)

이 앱은 본질적으로 오디오 제품이므로 별도 설계가 필요하다.

**음성 — 캐싱형 TTS**
- 코칭 멘트는 유한한 문구 풀에서 나오므로, **실시간 TTS 호출이 필요 없다.**
- 고품질 뉴럴 TTS(클로바·구글 등)로 문구 풀을 **사전 1회 합성 → 음성파일로 캐싱 → 재생만.**
- 효과: ①비용은 문구당 1회만 (재생 무료) ②뉴럴 TTS급 음질 ③오프라인 재생까지 동시 해결.
- 폰 내장 TTS는 무료지만 한국어 음질이 로봇 같아 "메이트" 콘셉트를 깎아먹는다 — 캐싱형으로 회피.
- 종료 후 요약 리포트만 매번 새로 생성되므로, 이건 텍스트로 두거나 내장 TTS로 처리.
- 페르소나마다 다른 TTS 목소리 부여 가능. 단 페르소나가 늘면 사전 합성·저장할 음성 세트도 함께 는다.
- **★음질 업그레이드 경로 (1차 현장 반영):** ①클로바 Pro(NeuVis)·타입캐스트·ElevenLabs 등 표현력 좋은 엔진으로 상향, 또는 ②자주 나오는 핵심 멘트 20~30개만 **실제 성우 녹음**하는 하이브리드. 캐싱 구조라 핵심만 녹음해도 "특별함" 체감이 크게 오른다.

**음악 — 케이던스 동기화 BGM**
- 저작권: 실제 곡은 라이선스 비용 큼. **로열티프리 인스트루멘탈 또는 직접 생성한 루프형 BGM** 사용 → BPM도 자유롭게 통제 (다양한 BPM 스템을 AI로 직접 생성하는 경로도 가능).
- **덕킹**: 메이트가 말할 때 BGM이 잠깐 줄었다 멘트 끝나면 복귀 → "코칭 없는 구간에 깔린다"는 의도와 정확히 일치.
- **BPM 전략**: 현재 케이던스에 맞추는 게 아니라, **현재보다 약간 높은 BPM**으로 깔아 케이던스를 자연스럽게 끌어올린다 (초보는 보폭 넓고 케이던스 낮은 경향). 즉 BGM이 배경음이 아니라 코칭 도구.
- 전환 타이밍: 케이던스가 흔들릴 때마다 곡을 바꾸면 거슬리므로, **코칭 사이나 km 구간 같은 자연스러운 지점에서만** 전환.

## 이중 축 설계 — 페르소나 × 레벨

두 개념을 **서로 다른 축**으로 분리해 설계한다. 합치면 나중에 구조를 갈아엎어야 한다.

**페르소나 (어떻게 말하는가) — 톤의 축**
- 한국어는 페르소나 차이가 주로 **문장 어미**에서 갈린다: 코치형 "~해봐요" / 친구형 "~하자, 같이 가자" / 동기부여형 "가자! 할 수 있어!"
- 데이터 구조를 **`상황 × 페르소나 → 멘트 변형`** 으로 짠다. 코치형만 출시해도 슬롯을 비워두면, 페르소나 추가가 *데이터만 채우는 작업*이 된다.

**레벨 (무엇을 향해 코칭하는가) — 로직의 축**
- 레벨은 톤이 아니라 **코칭 로직 자체**가 바뀐다. 페르소나와 독립.
- **초보 모드**: 관대함. 페이스 숫자에 집착 X, "천천히 가도 괜찮아요", 케이던스·부상예방 중심, run/walk 인터벌 고려.
- **중급 모드**: 깐깐함. 스플릿, 목표 레이스 페이스, 네거티브 스플릿, 경사 보정 페이스(GAP).
- 둘은 거의 정반대 성향 → 하나로 합치지 말고 별도 설정. 같은 "코치형 톤"이라도 초보냐 중급이냐에 따라 *조언 내용*이 완전히 달라야 한다.

> 정리: 페르소나는 **말투**, 레벨은 **내용**. (코치형 × 초보) (코치형 × 중급) (친구형 × 초보) … 가 모두 가능해야 한다.

## 기술 리스크 메모

1. **실시간 루프에 생성형 AI를 매번 호출하지 말 것.** 뛰는 중엔 네트워크 끊김·지연·배터리 문제가 발생한다. 실시간 코칭은 **가벼운 규칙 + 사전 생성한 음성 문구 풀**로 폰 안에서 처리하고, 생성형 AI는 ①문구 풀을 미리 다양하게 만들기 ②종료 후 서사형 리포트, 두 군데에 집중시킨다.
2. **★GPS 페이스 노이즈 (1차 현장 검증).** 계단·수직이동·고층 사이에서 순간 페이스가 급등한다. 평균으로 누르면 지연(=적막), 원시값을 쓰면 오발사. **중앙값(median) 필터 + 정확도(accuracy) 낮은 점 버리기 + 비현실적 속도 컷**으로 처리하고, 경사 보정·케이던스·기압계를 보조 신호로 쓴다.
3. **케이던스는 GPS 페이스보다 빠르고 덜 튀는 신호.** 계단에서도 케이던스는 엉뚱하게 안 뛴다. 반응성이 필요한 코칭은 케이던스를 우선 신호로 고려.
4. **GPS 음영지역**(산·터널·고층 사이) 대응을 초기에 설계에 반영. 가속도계로 보간하는 방식이 필요하다.
5. **배터리**가 곧 사용성. 센서 폴링 주기와 화면 갱신 주기를 처음부터 보수적으로 잡는다.
6. **보폭은 GPS가 아니라 가속도계+케이던스 조합**으로 계산하는 게 정확하다. 역할 분담을 명확히.
7. **★음성 자연스러움이 "특별한 경험"의 관문 (1차 현장 검증).** 동행자 느낌까진 도달했으나 특별함엔 미달. 캐싱형이라 자주 나오는 핵심 멘트만 골라 품질을 올리는 게 효율적.

## 결정 사항 (확정)

- **음성**: 합성 TTS, 사전 합성 후 음성파일 캐싱 방식 (비용·오프라인 동시 해결).
- **페르소나**: 기본값은 차분한 코치형. 추후 친구형·동기부여형으로 확장 (어투·톤·단어 다양성). 구조는 처음부터 다중 페르소나 대비.
- **타깃**: 초보 러너 우선. 추후 중급자도 떠나지 않도록 기록 단축·기록용 중급 모드로 확장.
- **음악**: 음악 취향 연계보다, 코칭 없는 구간에 깔리는 케이던스 동기화 BGM. 덕킹 처리.

## 남은 열린 질문

- TTS 보이스 **벤더 선정** (클로바 vs 구글 등) — 한국어 음질·가격·페르소나용 보이스 다양성 비교 필요.
- BGM을 **로열티프리 라이브러리**로 갈지, **직접 생성한 루프형 스템**으로 갈지 (후자는 BPM 통제·차별성↑, 제작 공수↑).
- 초보→중급 **전환 트리거**: 사용자가 수동 선택인가, 기록을 보고 앱이 제안하는가?
- 페르소나별로 **TTS 목소리도 다르게** 줄지 (다양성↑, 사전 합성 음성 세트도 배수로 증가).
